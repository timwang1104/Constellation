# PRD：prj_kanban（面向 AI Agent 的开发任务看板与依赖约束）

## 1. 背景与问题

在软件开发项目中，任务拆解后往往需要被不同执行者（人类或 AI Agent）认领与推进。传统看板擅长表达“状态流转”，但对以下场景支持不足：

- 任务无人认领时，AI Agent 无法基于规则主动认领并推进
- 任务存在前置约束（依赖）时，后续任务不应被提前认领
- 人类需要同时理解“项目任务推进情况”和“系统本身的架构形态”，但两者是不同视角

本项目 prj_kanban 的目标是提供一个“任务看板 + 任务依赖图 + Agent 认领/执行框架”的统一系统，用于管理 AI Agent 参与的软件开发项目。

## 2. 目标与非目标

### 2.1 目标

- 支持通过看板管理开发任务（创建/拆分/状态流转/负责人）
- 支持任务间依赖约束（前置任务未完成时，后续任务不可认领）
- 支持 AI Agent 监测无人认领任务并按策略主动认领
- 支持人类以不同视图理解项目：任务推进视图与系统架构视图
- 保证多人/多 Agent 并发认领时的数据一致性与可追溯性

### 2.2 非目标（第一阶段不做）

- 不做通用项目管理全家桶（成本、工时、预算、复杂报表）
- 不做强绑定某一代码托管平台（可先适配一种，后续扩展）
- 不做复杂流程建模（如 BPMN/Petri 网的全量能力）

## 3. 用户与使用场景

### 3.1 用户角色

- 项目负责人（Owner）：定义任务、依赖关系、策略与权限
- 开发者（Human Dev）：查看看板、手动认领/执行任务、Review Agent 产出
- AI Agent（Executor）：自动发现可认领任务并执行，回写进度与结果
- 审计/观察者（Observer）：只读查看项目状态与日志

### 3.2 典型场景

1) 负责人创建任务与依赖：A（搭建脚手架）→ B（实现登录）→ C（联调）
2) 看板中出现无人认领任务 A，Agent 自动认领 A 并执行
3) B 因依赖 A 未完成而被标记为“被阻塞”，Agent 不可认领
4) A 完成后，B 变为“可认领”，Agent 或人类可继续推进
5) 人类需要了解系统整体实现方式与边界，查看架构视图与关键流程

## 4. 核心概念与模型

### 4.1 任务（Task）

任务是最小可管理单元，包含：

- 标题、描述、标签、优先级、预计规模（可选）
- 状态：待办 / 进行中 / 已完成 / 已阻塞 / 已取消（可扩展）
- 负责人：人类用户或 Agent（可为空）
- 任务产出物链接：PR、Commit、构建产物、文档、工单等（可选）

### 4.2 依赖（Dependency）与任务图

- 使用“任务依赖图”表示前置约束，模型为 DAG（有向无环图）
- 边 A → B 表示 B 依赖 A（A 未完成时，B 不可认领）
- 系统需要提供环检测：不允许创建导致环的依赖关系

### 4.3 认领（Claim）

- 认领是“将任务负责人从空/其他执行者变更为当前执行者”的动作
- 认领必须是原子操作，避免并发下重复认领
- 认领前需校验任务可认领条件（见 5.2）

## 5. 功能需求

### 5.1 看板管理（面向人类）

- FR-1：支持创建/编辑/归档任务
- FR-2：支持看板列配置（默认：ToDo / Doing / Done）
- FR-3：支持拖拽或操作将任务在列间流转，并记录变更历史
- FR-4：支持按负责人/标签/状态/阻塞原因筛选与搜索

### 5.2 任务可认领规则（约束）

- FR-5：当任务存在任意未完成前置任务时，任务不可被认领
- FR-6：当任务已被其他执行者认领且未释放时，不可被再次认领
- FR-7：任务被手动标记为“阻塞”时，不可被认领（除非解除阻塞）
- FR-8：可配置“仅允许特定角色/特定 Agent 认领某类任务”的策略（可选）

### 5.3 任务依赖维护（DAG）

- FR-9：支持在任务间添加/删除依赖关系（A → B）
- FR-10：添加依赖时进行环检测，若形成环则拒绝并提示路径
- FR-11：在 UI 中展示任务的阻塞信息：阻塞来源任务列表

### 5.4 Agent 自动认领（核心）

- FR-12：Agent 可周期性/事件驱动扫描“无人认领且可认领”的任务
- FR-13：Agent 认领采用策略引擎排序（默认：优先级高、关键路径优先、最少阻塞风险）
- FR-14：Agent 认领成功后进入执行流程，持续回写任务进度与产出物链接
- FR-15：Agent 执行失败时可回写失败原因并支持重试/转交/回滚状态

### 5.5 视图与可视化

- FR-16：提供看板视图（状态流转为主）
- FR-17：提供任务依赖图视图（DAG，支持缩放、展开上下游、定位阻塞源）
- FR-18：在看板卡片上以轻量标记呈现依赖状态（如：被阻塞/阻塞他人）
- FR-19：提供项目架构视图（建议采用 C4：Context/Container/Component 三层）
- FR-20：提供“任务 ↔ 架构元素”的关联：任务可关联到模块/服务/组件（用于影响面分析与导航）

### 5.6 审计与可追溯性

- FR-21：记录关键事件：创建任务、状态变更、依赖变更、认领/释放、执行日志摘要
- FR-22：支持按任务查询完整时间线与执行者（人类/Agent）

### 5.7 Agent 会话、进度与对话记录

- FR-23：当 Agent 开始执行任务时，创建可追踪的 Agent 会话（Session），并与任务关联
- FR-24：Agent 会话支持进度上报（步骤/阶段/百分比/当前动作），并可被人类用户实时查看
- FR-25：系统保存 Agent 对话与工具调用记录，并支持按任务/会话查询与导出
- FR-26：任务完成后支持对会话记录进行归档（包含关键事件与对话记录），并生成可引用的归档条目
- FR-27：任务完成且归档成功后，支持按策略销毁 Agent 会话（停止运行、释放资源、清理临时数据）
- FR-28：会话记录保留期可配置；归档后支持对热数据进行裁剪/删除，同时保留最小审计摘要

## 6. 关键流程（文字版）

### 6.1 Agent 发现并认领任务

1) 监听事件或定时扫描任务集合
2) 过滤：负责人为空、状态允许、未阻塞
3) 校验依赖：所有前置任务已完成
4) 排序：应用认领策略
5) 原子认领：写入负责人=Agent，并记录认领事件
6) 执行：生成/运行任务计划，回写进度与产出物
7) 完成：更新为已完成并解除后续任务的阻塞

### 6.2 依赖变更校验

1) 用户发起添加依赖 A → B
2) 系统检查：A 与 B 存在、权限允许
3) 环检测：若从 B 可达 A，则拒绝
4) 持久化依赖边并更新相关任务阻塞状态

## 7. 权限与安全

- 负责人可配置：谁能创建任务、谁能配置依赖、谁能配置 Agent 策略
- Agent 只应拥有完成任务所需的最小权限（例如：写任务状态/日志、读任务与依赖）
- 所有敏感信息（API Key、Token）必须通过安全配置管理，不进入任务描述与日志
- Agent 对话与工具调用记录需支持脱敏与访问控制，避免泄露密钥、个人信息与内部链接

## 8. 非功能需求

- NFR-1：并发一致性：同一任务不可被多个执行者同时认领（强一致/事务/乐观锁均可）
- NFR-2：可用性：任务列表与认领流程在常规负载下保持稳定响应
- NFR-3：可观测性：关键流程具备指标与日志（认领成功率、失败原因分布、阻塞任务数量）
- NFR-4：扩展性：支持增加新的任务状态、认领策略、外部集成（Git/CI/IM）
- NFR-5：数据治理：会话记录与归档产物支持配置化保留期与清理策略，并可审计

## 9. 数据与接口（概念级）

### 9.1 主要实体

- Task：任务主体
- DependencyEdge：依赖边（from_task_id, to_task_id）
- ClaimRecord：认领记录（task_id, assignee, time, reason）
- TaskEvent：事件流水（用于审计与时间线）
- ArtifactLink：产出物链接（可选）
- AgentSession：Agent 会话（与任务关联，承载进度与记录）
- AgentMessage：Agent 对话与工具调用记录（可选）
- ArchiveRecord：会话归档记录（指向归档产物与摘要信息）（可选）

### 9.2 核心接口（示例）

- 创建/更新任务：POST/PUT /tasks
- 认领任务（原子）：POST /tasks/{id}/claim
- 释放任务：POST /tasks/{id}/release
- 更新状态：POST /tasks/{id}/transition
- 维护依赖：POST/DELETE /tasks/{id}/dependencies
- 获取依赖图：GET /projects/{id}/task-graph

## 10. 验收标准（第一阶段）

- AC-1：当任务存在未完成前置任务时，任何执行者调用认领接口均失败，并返回可读错误信息
- AC-2：当任务无人认领且依赖满足时，Agent 能在一次扫描周期内成功认领至少一个任务
- AC-3：并发下对同一任务发起两次认领，最多一次成功（另一方得到冲突/已被认领提示）
- AC-4：依赖关系新增导致环时被拒绝，并返回环路径或参与节点信息
- AC-5：看板视图可展示任务状态与负责人；依赖视图可展示上下游；两者可以互相导航到同一任务详情
- AC-6：任务可关联架构元素，并能从架构元素反查相关任务列表
- AC-7：当 Agent 执行任务时，人类用户可查看该任务的会话与进度更新
- AC-8：任务完成后可导出该任务的对话与关键事件记录，并具备脱敏处理
- AC-9：任务归档后，系统可按保留策略清理热数据，同时保留可追溯的审计摘要

## 11. 后续演进方向（可选）

- 引入关键路径计算（CPM）作为默认优先级策略的一部分
- 支持更复杂流程（条件网关、资源互斥）时引入 BPMN/Petri 网能力
- 与代码仓库/CI 深度联动：PR 状态驱动任务状态、自动生成产出物链接
